# Resource Mappings — Cluster-Aware Passthrough (Proxmox VE ≥ 8)

> **TL;DR**: On a single Proxmox host, `hostpci0: 0000:03:00,pcie=1` is fine. In a **cluster** with HA or migration, the physical BDF of the same card differs per node — which breaks VM config. Proxmox VE 8 introduced **Resource Mappings**: a cluster-wide logical name (e.g. `arc-a310`) that resolves to a different physical device per node. All examples in this repo use physical BDFs because the validation host is a standalone node; this doc shows how to switch to mappings when you cluster.

## The Problem

All examples in this repo (and the [Quick Start](../README.md)) use physical BDFs:

```
hostpci0: 0000:03:00,pcie=1
```

That's fine on a single host. In a cluster it breaks:

- On `node-a` the Intel Arc A310 sits at `0000:03:00.0`
- On `node-b` the same card model sits at `0000:04:00.0` (different slot topology, different enumeration)

A VM config written against a physical BDF becomes **node-specific**. Live-migration fails because the target node has no device at the hardcoded address. HA failover starts the VM on `node-b` with a config that points into the void — QEMU aborts with `Could not assign device 0000:03:00.0, error -22`.

## The Fix: Logical Mapping Names

A Resource Mapping is a cluster-wide logical name plus per-node entries resolving it to a local BDF. The VM config references only the name:

```
hostpci0: mapping=arc-a310,pcie=1
```

At VM start on any node, Proxmox resolves `arc-a310` → the correct local BDF automatically.

## Setup via `pvesh`

The primary CLI is Proxmox's API helper `pvesh`. Example for an Intel Arc A310 present on two nodes at different BDFs:

```bash
pvesh create /cluster/mapping/pci --id arc-a310 \
  --description "Intel Arc A310 for Win11 passthrough" \
  --map id=8086:56a6,node=node-a,path=0000:03:00.0,subsystem-id=1849:6007 \
  --map id=8086:56a6,node=node-b,path=0000:04:00.0,subsystem-id=1849:6007
```

Field breakdown:

| Field | Purpose |
|-------|---------|
| `--id NAME` | Cluster-wide logical name (used later in `qm set`) |
| `id=VENDOR:DEVICE` | PCI vendor:device IDs from `lspci -nn` — same on all nodes (hardware fingerprint) |
| `node=NODENAME` | Proxmox node where this physical device lives |
| `path=0000:BB:DD.F` | Physical BDF **on that specific node** (differs per node) |
| `subsystem-id=VENDOR:DEVICE` | Optional but recommended: subsystem IDs from `lspci -vnn` (protects against accidental binding to a visually similar but distinct card, e.g. a different vendor's rebadge of the same chip) |

Find the values you need on each node:

```bash
# On each cluster node:
lspci -nn | grep -i -E 'VGA|Display|3D controller|Audio'   # get VENDOR:DEVICE + BDF
lspci -vnn -s 03:00.0 | grep Subsystem                      # get subsystem vendor:device
```

Add another node to an existing mapping:

```bash
pvesh set /cluster/mapping/pci/arc-a310 \
  --map id=8086:56a6,node=node-c,path=0000:05:00.0,subsystem-id=1849:6007
```

List all mappings / inspect one:

```bash
pvesh get /cluster/mapping/pci
pvesh get /cluster/mapping/pci/arc-a310
```

### USB Mappings

For USB devices (license dongles, TV tuners, serial adapters), the analogous command uses vendor/product ID:

```bash
pvesh create /cluster/mapping/usb --id license-dongle \
  --description "Vendor Foo licence dongle for app bar" \
  --map id=1234:5678,node=node-a \
  --map id=1234:5678,node=node-b
```

USB mappings identify by vendor/product ID (stable across reboots and port changes). For per-port mapping, append `,path=<bus>-<port>` — less robust because port numbering can shift after reboots or cable moves; prefer vendor/product identification whenever possible.

## Binding the Mapping to a VM

Same flags as for physical BDFs (`pcie=1`, `x-vga=1`, `rombar=1`, etc.) still apply — just replace the BDF with `mapping=NAME`:

```bash
qm set <vmid> --hostpci0 mapping=arc-a310,pcie=1
qm set <vmid> --hostpci1 mapping=arc-a310-audio,pcie=1
```

The vendor-specific QEMU `args:` line (generated by `scripts/generate-vm-args.sh`) is independent of how the device is referenced — mapping or physical BDF doesn't matter for `-cpu host,kvm=off,…`. Combine freely.

## Live Migration

Resource Mappings alone don't make a passthrough VM live-migratable. The device additionally has to be flagged **Live Migration Capable**, which is really a vendor-driver statement: *yes, this device supports transparent migration*. For most consumer GPUs the answer is **no** — the driver loses PCI config-space state at migration time.

Without the flag:
- **HA failover** → short downtime (stop on source, start on target). Usually acceptable; certainly better than pinning the VM forever.
- **Planned offline migration** → same: stop VM, migrate config, start on target. Works as long as the mapping resolves on the target.

Setting the flag:
- **GUI**: Datacenter → Resource Mappings → edit the mapping → check "Live Migration Capable".
- **API**: use `pvesh help /cluster/mapping/pci/<ID> set --verbose` on a Proxmox host to discover the exact parameter name for your version (it has varied slightly across 8.x releases).

## When to Use This (and When Not)

| Scenario | Use Resource Mappings? |
|----------|------------------------|
| Single Proxmox host | No — physical BDF is simpler, no benefit from the indirection |
| Proxmox cluster with HA, passthrough VMs | **Yes — mandatory for reliable failover** |
| Planned migration between nodes with different hardware layouts | **Yes** |
| Dedicated passthrough node that will never move | Optional — indirection buys nothing here |
| Hardware replacement (new card in different slot on same node) | Mapping lets you update the device registration once without touching any VM config — **convenient** |

All recipes in this repo use physical BDFs because the validation host is standalone. When you extend to a cluster, replace every `hostpci<N>: 0000:XX:00,pcie=1` in the VM config with `hostpci<N>: mapping=NAME,pcie=1` **after** registering the mapping cluster-wide.
